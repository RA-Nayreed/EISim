\section{Executive Summary}
%===============================================================================

This document presents a rigorous comparative analysis of edge orchestration simulation results between homogeneous (baseline) and heterogeneous server configurations. The evaluation examines the impact of server capacity heterogeneity on DRL-based pricing performance across three control topologies: Centralized, Hybrid, and Decentralized. All comparisons use $n=5$ simulation runs for statistical validity.

\subsection{Key Findings}

\begin{enumerate}
    \item \textbf{Moderate Profit Improvements}: Heterogeneous configurations show modest profit improvements:
    \begin{itemize}
        \item \textbf{20 Servers}: Average change of $-5.7\%$ across all topologies (topology-dependent)
        \item \textbf{100 Servers}: Average change of $+11.4\%$ across all topologies
    \end{itemize}

    \item \textbf{Topology-Dependent Effects}: Impact varies significantly by topology:
    \begin{itemize}
        \item \textbf{Centralized}: Consistent improvement (+3--6\% for 20 servers, +11--18\% for 100 servers)
        \item \textbf{Hybrid}: Mixed results (-58\% to +11\% for 20 servers, +13--22\% for 100 servers)
        \item \textbf{Decentralized}: Mixed results for 20 servers (-35\% to +11\%), and for 100 servers (-15\% to +14\%)
    \end{itemize}

    \item \textbf{Topology Ranking Partially Stable}: Rankings show moderate stability---2/4 device counts match for 20-server scenario (50\%), 3/4 match for 100-server scenario (75\%). Centralized consistently remains the best-performing topology.

    \item \textbf{Offload Rate Changes}: Heterogeneous configuration significantly increases task offload rates for Hybrid and Decentralized topologies at lower device counts (+47\% to +223\% for 1K devices), while Centralized remains stable with minimal change.

    \item \textbf{Energy Efficiency}: Heterogeneous configurations show modest energy consumption changes, with Centralized topology achieving consistent $\sim$2.5\% reduction.
\end{enumerate}

\section{Experimental Setup}
%===============================================================================

\subsection{Simulation Environment}

The evaluation employs EISim, a discrete-event simulator for edge computing environments. Simulations were conducted with:
\begin{itemize}
    \item \textbf{Duration}: 500 seconds per episode
    \item \textbf{Price Slot Duration}: 5 seconds
    \item \textbf{Edge Device Counts}: 1000, 2000, 3000, 4000
    \item \textbf{Server Counts}: 20 (high-capacity), 100 (distributed)
    \item \textbf{Control Topologies}: Centralized, Hybrid, Decentralized
    \item \textbf{Simulation Runs}: $n=5$ per configuration (for statistical validity)
\end{itemize}

\section{DDPG Model Architecture}
%===============================================================================

\subsection{State Space Design}

The pricing agent employs a Deep Deterministic Policy Gradient (DDPG) architecture with a state space that differs between configurations:

\textbf{Homogeneous Configuration (2-dimensional state):}
\begin{equation}
s_t^{homo} = [\bar{q}_t, \bar{\lambda}_t]
\end{equation}
where:
\begin{itemize}
    \item $\bar{q}_t$: Average queue length across cluster members (scaled)
    \item $\bar{\lambda}_t$: Average task arrival rate over the price slot (scaled)
\end{itemize}

\textbf{Heterogeneous Configuration (3-dimensional state):}
\begin{equation}
s_t^{hetero} = [\bar{q}_t, \bar{\lambda}_t, c_{norm}]
\end{equation}
where $c_{norm}$ represents the normalized cluster capacity:
\begin{equation}
c_{norm} = \frac{\sum_{i \in cluster} MIPS_i}{\bar{MIPS}_{scenario}}
\end{equation}

This additional dimension enables the agent to incorporate capacity awareness into pricing decisions.

\subsection{Network Architecture}

\textbf{Actor Network:} Input $\rightarrow$ Dense(64, ReLU) $\rightarrow$ Dense(64, ReLU) $\rightarrow$ Dense(1, Tanh) $\rightarrow$ scaled to $[0, 1]$

\textbf{Critic Network:} Input(state+action) $\rightarrow$ Dense(64, ReLU) $\rightarrow$ Dense(64, ReLU) $\rightarrow$ Dense(1, Linear)

\textbf{Training:} AMSGrad optimizer, $\alpha_{actor}=10^{-4}$, $\alpha_{critic}=10^{-3}$, $\gamma=0.99$, $\tau=0.005$, batch size 64, replay buffer 10,000.

\section{Conclusions}
%===============================================================================

\begin{enumerate}
    \item \textbf{Fair comparison established}: Total MIPS, RAM, and storage are exactly equal (6M MIPS, 1.6M MB RAM, 25.6M MB storage) between homo and hetero configurations, with $n=5$ runs ensuring statistical validity.

    \item \textbf{Moderate profit improvement}: With proper baselines ($n=5$), heterogeneous configuration shows:
    \begin{itemize}
        \item 20 Servers: $-5.7\%$ average (topology-dependent, ranging from $-58\%$ to $+11\%$)
        \item 100 Servers: $+11.4\%$ average across all topologies (ranging from $-15\%$ to $+22\%$)
    \end{itemize}

    \item \textbf{Topology-specific effects}: Centralized topology benefits most consistently from heterogeneous configuration, showing improved profit and QoS. Hybrid and Decentralized topologies show profit improvement but reduced edge success rates.

    \item \textbf{Rankings partially stable}: 50\% of topology rankings match between configurations for 20-server scenario, 75\% for 100-server scenario. Centralized consistently remains the best performer.

    \item \textbf{3D state space effectiveness}: Adding normalized capacity as a state dimension provides moderate benefits, primarily for Centralized topology at 100-server scale.

    \item \textbf{Scale-dependent workload distribution}: Capacity-aware pricing exhibits different behavior across scales---favoring high-capacity servers at 20-server scale (A:C task ratio 4.3:1 vs capacity 3.3:1) while achieving load-balancing at 100-server scale (A:C task ratio 1.2:1 vs capacity 2.0:1).
\end{enumerate}
